User-first design

  • As a new user, I can create an account
    • John visits the application's home page
    • He clicks on the register button
    • The website shows a sign up form, and he fills it in with his e-mail, username, and password
    • He clicks on the sign up button
    • He receives an e-mail asking for him to click on the link to confirm his account
    • He clicks on the link, and his account is finally ready to use
  • As a normal user, I can log into my account
    • John visits the application's home page
    • He clicks on the log in button
    • The website shows a log in form, and he fills in his username/e-mail and password
    • He clicks on the log in button
    • The user is now logged and the website shows the default page
  • As a normal user, I can post a message
    • John is logged in and in the application's main page
    • He clicks on the new message button and a form to type in the message appears on the screen
    • He types in the message and then hits the post button
    • The form disappears from the screen and the message can now be seen on John's profile
  • As a normal user, I can see my mentions
    • John is logged in and in the application's main page
    • He clicks on the mentions tab
    • The website shows a list of all the messages in which John has been mentioned in from the most recent one to the oldest one
  • As a normal user, I can see the messages of the users I follow
    • John is logged in and in the application's main page
    • By deafult, all the messages of the people he follows appear on this page
    • He can scroll down to see the older messages, as they are ordered from most recent to oldest
  • As a normal user, I can search for someone
    • John is logged in and in the application's main page
    • He clicks on the search bar and types in a username
    • A list of users that match the criteria of the search appears while John types in the username
  • As a normal user, I can follow someone
    • John is logged in and in the application's main page
    • He searches for the user he wants to follow
    • When the desired user appears on the dynamic list, he clicks on it
    • The website now shows the profile of the desired person
    • He clicks on the follow button
  • As a normal user, I can see one of my followers' messages
    • John is logged in and in the application's main page
    • He goes to his profile
    • He clicks on the following list
    • The website displays a list of all the users he is following
    • He searches for the desired user, and when he finds it, he clicks on it
    • The website shows the user's profile and all the messages he has posted can be seen from newest to oldest
  • As a normal user, I can check someone's following list
    • John searches for a user as indicated in a previous bullet point
    • He clicks on the desired user and the website shows that user's profile
    • John clicks on the following list and the website displays a list of all the users followed by user selected
  • As a normal user, I can check someone's followers list
    • It's the same use case as the previously mentioned but, instead of clicking on the following list, John clicks on the followers list; and the website displays the followers of that user
  • As a normal user, I can edit my profile information
    • John is logged in and in the application's main page
    • He clicks on the edit profile button
    • The website shows the form with the information about the user
    • He edits the information and then hits the save button
  • As a normal user, I can change my profile picture
    • It's the same case as the previous one, but insteaed of editing the information and hitting the save button, the user clicks on the change picture button
    • The website shows a pop up where the John can browse to look for the picture
    • Once he finds it, he double clicks on the picture
    • The pop up disappears, and a loading icon is shown on screen
    • Once the picture is uploaded, a success message is shown
  • As a normal user, I can see a list of recommended users to follow
    • John is logged in and in the application's main page
    • He clicks on the who to follow button
    • The website shows a list of users that John might like to follow
  • As a normal user, I can search for all the messages that have a certain hashtag in them
    • John is logged in and in the application's main page
    • He clicks on the search bar, and types a hashtag (#) followed by the keyword he wants to search by
    • Once he is done, he hits enter
    • The website shows a page with all the messages that have the keyword that was looked up
  • As a normal user, I can log out of my account
    • John is logged in and in the application's main page
    • He clicks on the log out button
    • The website displays a confirmation dialog that asks the user whether they're sure they want to log out
    • He hits on the yes button
    • The website logs out the user, and shows the home page

Data-first design

The following resources are needed to develop the social media app:

  • Users
    • E-mail
    • Username
    • Password
    • Picture
  • Messages
    • Date
    • Text
    • Reference to the User who posted the message
  • Following
    • Reference to a User (the one that follows)
    • Reference to a User (the one that is being followed)

    This structure represents the N-N relationship between Users. A User can follow multiple Users, and can be followed by multiple Users. Having this table makes it easy to search for the following and follower's list of a particular User.

  • Mentions
    • Reference to a Message
    • Reference to a User

    This structure represents the N-N relationship between Users and Messages. N Users can be mentioned in a Message, while a User can be mentioned in N Messages. This kind of structure makes searching for a user's mentions fast.

  • Keywords
    • String
  • Hashtags
    • Reference to a Message
    • Reference to a Keyword

    This structure represents the N-N relationship between Hashtags and Messages. A Message can contain N Hashtags, while a Hashtag can be in N Messages. This structure helps to retrieve the messages that contain a specific hashtag fast. Instead of having a separate Keywords table, the String could be directly stored in this table. The disadvantage of doing that, is that the same string is going to be stored over and over again (instead of a simple id).
